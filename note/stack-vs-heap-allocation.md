# Rust 栈分配与堆分配的本质

> Rust 不是"默认在栈上"或"默认在堆上"，而是**编译期能确定大小的就放栈上，确定不了的就放堆上**。

---

## 核心规则

Rust 在编译期决定变量放在栈还是堆上：

| 编译期能确定大小（Sized） | 编译期不确定大小（?Sized） |
|---|---|
| 放在**栈上** | 放在**堆上**（或间接访问） |
| `i32`, `bool`, `char`, `(i32, u8)` | `String` 的数据内容 |
| `[T; N]`（定长数组） | `Vec<T>` 的数据内容 |
| `&T`, `Box<T>`, `struct` | `str`, `[T]`, `dyn Trait`（必须通过间接方式使用） |

---

## 内存分配全景

```rust
fn main() {
    // 栈上分配
    let x: i32 = 42;                    // i32 大小固定，直接放栈
    let arr: [u8; 5] = [1, 2, 3, 4, 5]; // 定长数组，编译期已知大小

    // 堆上分配（数据在堆上，元数据在栈上）
    let v = vec![1, 2, 3, 4, 5];       // Vec 长度运行时才确定
    let s = String::from("hello");     // String 内容运行时才确定
}
```


## 感悟

Rust 本身也是有栈上分配的。**所有在编译期确定大小的类型都会分配到栈上，不确定的分配到堆中。** 这不是"智能优化"，而是 Rust 类型系统的自然产物：

- `Sized` trait 是隐式的，所有泛型默认 `T: Sized`
- 编译期能确定大小的类型 → 栈上
- 编译期不能确定大小的类型（DST） → 必须通过胖指针或间接方式访问堆上数据

所以 `String` 的结构体本身在栈上（ptr + len + cap），但它指向的字符串内容在堆上。`&str` 本身在栈上（ptr + len），但它指向的数据可以在任何位置（只读段、堆、栈）。

---

"`★ Insight ─────────────────────────────────────`
1. **栈 vs 堆不是选择，而是必然**：编译期确定大小 → 栈；不确定 → 堆，这是类型系统的自然结果
2. **Rust 不像 Go 有逃逸分析**：Go 在编译期决定"这个变量要不要逃逸到堆"；Rust 的类型本身已经决定了分配位置
3. **胖指针是栈和堆的桥梁**：它本身在栈上（Sized，16 字节），通过 ptr 指向可能在不同位置的数据
`─────────────────────────────────────────────────`"
